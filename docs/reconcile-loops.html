Pull vs Push · Reconcile Loop

ArgoCD 는 안에서 돈다.
Ansible 은 밖에서 민다.

같은 GitOps 4속성 중 셋(선언·SSOT·롤백)은 동일하다. 갈리는 건 네 번째 하나 — reconcile 루프가 대상의 안에 있느냐, 밖에 있느냐. 그 위치 차이가 자가치유와 재난 복구를 가른다.

자동 신호 흐름 reconcile 펄스 사람 / 끊기는 지점 경계·의존

ArgoCD — pull

컨트롤러가 대상 안에 상주

닫힌 고리. live 가 컨트롤러로 되돌아오는 백엣지가 루프를 대상 안에서 닫는다. 누가 라이브를 망가뜨려도 스스로 수렴한다 — 사람·외부 트리거 불필요.

OpenWrt — push

엔진이 대상 밖에, 게다가 하류에

열린 선. 되돌아오는 백엣지가 없다 — 외부 트리거(cron)가 주기적으로 밀어야 수렴. 게다가 러너는 대상 하류라, 대상이 죽으면 러너도 죽고 노트북 bare-run(=사람)으로 떨어진다.

사람이 환원 불가능한 지점

01 · 두 모델 공통

최초 의도

desired state 를 git 에 적는 건 언제나 사람이다. ArgoCD 도 마찬가지 — GitOps 가 자동화하는 건 "적힌 의도를 라이브에 반영하는 루프"지 의도 그 자체가 아니다.

02 · push 모델만의 추가

재난 복구

대상(OpenWrt)이 죽으면 그 위의 k3s·러너도 같이 죽는다 = 복구 도구가 복구 대상의 하류. 이때 마지막 폴백이 노트북 bare-run = 사람. ArgoCD 에는 없는, push 모델 고유의 환원 불가 지점.

평상시(happy path)엔 cron 스케줄러가 ArgoCD 트리거 루프를 대신 → 사람 개입 0. 사람은 ① 의도 작성과 ② 재난 복구에만 남는다.
